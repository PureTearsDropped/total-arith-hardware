### 利用者が 一周させた洞察: `1/0` こそ 符号不明の 原型

**利用者**: 「既存の `1/0 = +inf` っておかしいのでは？ 0 の符号が どっちか分からないのに」

**その通り**（実測）: `1/(+0.0)=+inf`, `1/(−0.0)=−inf` —— 答えが **0 の符号ビットに依存**するが、
`x−x` や 測定丸めの 0 は **符号が 事実として無い**。数学的にも `1/0` は 両側極限が違う（右+∞・左−∞）
⟹ **符号は存在しない**。IEEE の `+inf` は 片方を 勝手に選んでいる。

**さらに 利用者の refine**: 「だったら それは 0 じゃなく ±MIN では？ 同じ点ではないよね？」

**その通り。監査人の ④「1/0 は ±MAX + 符号不明」は 誤り。** 真の 0 と ±MIN は **別の点**（§7.6.2）:

```
IEEE: 潰れた ±1e−400 も 真の 0（x−x）も **全部 0 に潰す** ⟹ 1/0 の符号が 分からず +inf を でっち上げ
この設計: **2 点を 分ける**:
    潰れ → ±MIN（符号あり）⟹ 1/±MIN = **±MAX（符号 決まる）**  ← 符号不明 すら 要らない
    真の 0            ⟹ 1/0 = **0**（Moore–Penrose・符号なし）  ← ∞ を返さない ⟹ 符号問題なし
```

> **IEEE が `1/0` の符号を でっち上げる 根本原因 = 「潰れた ±MIN」と「真の 0」を 区別しないこと。**
> **この設計は 朝いちばんに ±MIN で 2 点を分けた ⟹ `1/0` の符号問題は そもそも 発生しない。**
> ＝ 朝の `a/0=0`／`±MIN`（0 を予約語に）が、二日目の `1/0` 符号問題を 先回りで 解いていた。

### 実数値 計算試験（`numerical_test.py`）で また 測定ミス

直すと **③④ とも 0**。実装は 健全。

> ＝ 今日 繰り返した「値の側で 何かを 落として 偽の結論」の 変奏。**真値は 実際に演算した 入力から 作れ。**

### 利用者の改善: 潰さずに 代表値で計算＋方向保持（量的境界）

「MAX/MIN を その値として 計算して、`MAX/(MAX/2)` なら `≥2` とすればいい」。
**監査人の「非厳密入力の 積/商 → 一律 境界なし」は 潰しすぎ**だった。区間伝播に直すと

### 監査人の 雑な一般化: 「セデニオン積は 常に 境界なし」は 誤り
